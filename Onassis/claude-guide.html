タブ① Chat — 概要

Project と Artifacts：日常のチャットを「資産化」する2機能

Project＝チャットの「専用作業部屋」。関連資料（ナレッジ：プロジェクトにアップロードした参照ファイル群）とカスタム指示（そのプロジェクト内の全チャットに適用される指示文）を一度設定すれば、毎回の説明が不要になる。

Artifacts＝会話から生まれる「単体の成果物」。一定以上のまとまった文書・コード・図・Webアプリをチャット横の専用ウィンドウに表示し、会話で繰り返し編集できる。公開リンクで配布でき、他人の公開Artifactは「Customize（カスタマイズ）」で自分のコピーを作って改変できる。

できること

  • Project：資料アップロード／プロジェクト単位の指示設定／チャット整理／Team・Enterpriseでのプロジェクト共有／ナレッジが大きい場合のRAG自動拡張（有料プラン、最大約10倍）
  • Artifacts：文書・コード・HTML/Reactアプリ・SVG・Mermaid図の生成と反復編集／バージョン切替／公開（Publish）・組織内共有（Share）／他人の作品のCustomizeによるコピー改変／AI搭載アプリ化・MCP連携・永続ストレージ（一部は有料プラン）

できないこと（誤解しやすい点）

  • ProjectはPC内のフォルダを直接読まない。ファイルは手動アップロードが必要（直接操作はCoworkの領分）
  • ArtifactsはローカルPCにファイルを書き込まない。成果物はブラウザ上の表示＋ダウンロード
  • 他人があなたの公開Artifactを「Customize」しても、元のArtifactは変更されない（相手側にコピーが作られる）
  • Project共有はTeam/Enterpriseプランの機能。個人プランでは原則自分専用

タブ① Chat — 活用シナリオ 2/2

シナリオB：Artifactsで作った成果物を編集・反復し、配布する

課題新人向けツールの使い方ガイドを作りたいが、Word往復の修正に時間がかかり、配布後の更新も面倒。
→
打ち手ArtifactsでHTMLガイド（または簡易Webアプリ）を生成し、会話で修正を反復。完成版を公開リンクで配布。
→
成果物URL1つで全員が見られるガイド。受け取った側は「Customize」で自部署向けに改変したコピーも作れる。

操作手順

  1. 生成：「新人向けの〇〇操作ガイドを、目次付きHTMLのArtifactで作って」と依頼。
  2. 反復編集：「専門用語に注釈を」「手順4に注意書きを追加」と会話で修正。版を比較しながら仕上げる。
  3. 公開：Artifactウィンドウ右上の「公開」をクリック→発行されたリンクを共有。Team/Enterpriseでは組織内のみの共有が可能。配布をやめたいときは公開済みタブから取り下げ（unpublish）できる。
ここに実際のスクリーンショットを配置：Artifactの公開ダイアログ（リンク発行画面）

注意点・つまずきポイント・コツ（Chatタブ）

  • 短い回答はArtifactにならない：単体で意味を持つ、ある程度の分量のコンテンツが対象。短文なら通常の返答になる。
  • 公開＝そのバージョンのコードも公開：公開時は表示内容と関連コードが見える状態になる。機密情報を含む版を公開しない。
  • Projectナレッジは「厳選」が効く：関係資料を全部入れるより、ブランドガイド・要件・スタイル指針など5〜6点に絞る方が回答精度が安定しやすい。
  • 古い資料の放置に注意：ナレッジは自動更新されない。改訂のたびに差し替える運用ルールを決める。
  • エラーが出たら「Try fixing with Claude」：Artifactの実行エラーはボタン1つでエラー内容をClaudeに渡して修正を試せる（成功は保証されない）。
タブ① Chat — 演習 ⏱ 今日の10分

演習：Projectを1つ作り、その中でArtifactを1つ生成する

ゴール：「資料を覚えさせた部屋で、配れる成果物を作る」という基本動線を、自分の実データで1往復体験する。読むだけで終わらせないための最小タスク。

手順

  1. 素材を1つ選ぶ：直近1ヶ月で2回以上参照した資料（規程・仕様書・議事録など）を1つだけ選ぶ。複数入れない。
  2. Projectを作る：サイドバー「プロジェクト」→新規作成→選んだ資料をナレッジにアップロード→指示欄に1行だけ書く（例：「回答時は必ずナレッジの該当箇所を根拠として示す」）。
  3. プロジェクト内でArtifactを依頼：「この資料の要点を、新メンバー向けの1枚HTMLサマリーとしてArtifactで作って」→ 生成後に「見出しを1つ増やして」と修正指示を1回出す。

完了条件（2つチェックできたら終了）

  • プロジェクト内チャットが、アップロードした資料の中身を踏まえて回答している
  • 右側のArtifactが、修正指示で実際に更新された（バージョンが進んだ）

つまずいたら

  • 資料の内容を参照してくれない → ファイルが大きすぎる可能性。該当章だけ抜き出して再アップロード。
  • Artifactにならず本文に出力される → 「Artifactとして作って」と明示する。短すぎる依頼はArtifact化されない。
  • 指示が効いていない気がする → 指示はプロジェクト内の新規チャットから適用される。既存チャットではなく新しく開始する。

5分で終わったら追加の1手：作ったArtifactを「公開」してリンクを自分宛てに送り、別端末・別ブラウザで開けることを確認する。

タブ② Cowork — 概要

Cowork：デスクトップでPC作業を「丸ごと任せる」

Coworkは、Claude Codeと同じエージェント基盤（計画→実行→検証を自律的に繰り返す仕組み）を、ターミナル不要のClaude Desktopアプリ上で使えるようにした機能。プロンプトに1問1答で返すのではなく、「結果」を依頼して席を外し、完成した実ファイル（整形済み文書、整理されたフォルダ、Excel、PowerPoint等）を受け取る使い方が前提。

できること

  • ローカルファイルの直接読み書き（手動アップロード不要）
  • 複数ステップ作業の分解・サブエージェントによる並列実行
  • 数式入りExcel、スライド、整形文書などの成果物生成
  • 長時間タスクの継続実行（通常チャットのような会話タイムアウトに縛られない）
  • /schedule による定期タスク（例：毎朝のブリーフィング）
  • Cowork内プロジェクト：関連タスクをファイル・指示・メモリ付きのワークスペースに整理（ローカル保存）
  • Pro/Maxはモバイルからタスクを投げ、デスクトップ側で実行させることも可能

できないこと・前提（誤解しやすい点）

  • デスクトップアプリ必須（macOS/Windows）。Web・モバイル単体では動かない
  • 有料プラン限定（Pro/Max/Team/Enterprise）
  • アプリを閉じる・PCがスリープすると実行中タスクは停止する
  • セッション（チャット・成果物）の他者共有は不可。メモリはプロジェクト内のみ保持
  • 通常チャットより使用量（usage）消費が大きい。単純な質問はChatで行うべき
  • 実ファイルを変更できる＝リスクも実在。機密フォルダへのアクセス付与は避け、削除は明示許可制

シナリオB：散在するメモから報告資料を毎週自動生成（定期タスク）

課題週次報告のために、議事メモ・ToDo・下書きをかき集めて毎週金曜に資料化している。
→
打ち手Coworkプロジェクトにメモフォルダと様式の指示を常設し、/schedule で毎週金曜に自動実行。
→
成果物毎週、所定様式の週次報告ドキュメント（またはスライド）が指定フォルダに自動生成される。

操作手順

  1. Coworkでプロジェクトを作成し、メモ用フォルダを紐付け。指示欄に「様式：①今週の成果 ②課題 ③来週予定。各3点以内」と登録。
  2. 一度手動で実行して品質を確認：「今週のメモから週次報告を作成して」。出力を見て指示を微調整する。
  3. 定期化：タスク内で /schedule と入力（またはサイドバー「Scheduled」）し、毎週金曜17時に設定。以後はPCとアプリが起動していれば自動で実行される。

注意点・つまずきポイント・コツ（Cowork）

  • アクセス権は最小限に：タスクに必要なフォルダだけを渡す。財務情報など機密ファイルを含むフォルダの付与は避ける（公式の安全ガイドでも明記）。
  • スリープ＝停止：長時間タスクや定期タスクは、PCが起きていてアプリが開いていることが条件。電源設定を確認する。
  • 使用量の消費が速い：マルチステップ実行はトークン消費が大きい。関連作業は1セッションにまとめ、単純な質問はChat側で行う。
  • 削除系の操作は許可プロンプトが出る：ファイル完全削除はAllowを押すまで実行されない。慌てて承認せず対象を確認する。
  • 最初は「捨ててもよいフォルダ」で試す：実ファイルを変更できる機能のため、初回はテスト用フォルダで挙動を掴んでから本番データに使う。

演習：テスト用フォルダで「計画→実行→成果物」を1サイクル観察する

ゴール：本番データに触れる前に、Coworkが「どこまで自分で進め、どこで許可を求めるか」を安全な環境で体感する。原本には一切触れないのがルール。

手順

  1. テスト環境を作る：デスクトップに「cowork-test」フォルダを新規作成し、種類の異なるファイル5〜10個（PDF・画像・Excelなど）をコピーして入れる。原本は移動しない。
  2. タスクを投げる：Claude DesktopでCoworkタブに切り替え、cowork-testフォルダを指定して依頼：「このフォルダ内のファイルを種類別のサブフォルダに整理し、何をどこへ移したかの一覧を result.md として同じフォルダに保存して」。
  3. 計画を読んでから実行し、進捗を観察する：実行前に表示される計画に目を通す→走らせる→ステップ表示を眺める。余裕があれば途中で「画像はさらに年月別に分けて」と追加指示（ステアリング）を1回試す。

完了条件（2つチェックできたら終了）

  • 「計画提示→実行→ローカルに成果物」の一連の流れを自分の目で確認した
  • cowork-testフォルダに整理済みサブフォルダと result.md が生成されている

つまずいたら

  • Coworkタブが表示されない → デスクトップアプリが最新版か（claude.com/download）、有料プラン（Pro/Max/Team/Enterprise）でログインしているかを確認。
  • タスクが途中で止まった → アプリを閉じた／PCがスリープした可能性。電源設定を見直して再実行。
  • 想定外のファイルを触ろうとする → そのまま承認せず、対象フォルダの指定を狭めて依頼し直す。

5分で終わったら追加の1手：同じタスク内で /schedule と入力し、定期実行の設定画面がどこにあるかだけ確認しておく（設定までは不要）。

タブ③ Claude Code — 概要

Claude Code：ターミナルで動く開発エージェント

Claude Codeは、ターミナル（コマンドライン）上でリポジトリを直接読み書きするエージェント型コーディングツール。「この機能を追加して」「このバグを直して」と自然言語で依頼すると、関連コードの特定→修正→（テストがあれば）実行確認までを自律的に行う。ファイル編集・コマンド実行・Gitコミットまで実際に「手を動かす」点が、コードを提案するだけのチャットとの本質的な違い。

できること

  • 説明文からの機能実装、バグ修正、リファクタリング、テスト作成
  • コードベース全体の構造把握と質問応答（手動でファイルを渡す必要なし）
  • Git操作の会話化（差分確認・コミット・ブランチ作成・コンフリクト解消）
  • MCP連携でGoogle Drive・Figma・Jira等の外部情報も参照
  • パイプ・スクリプト化（claude -p "…"）によるCI/CD組み込み
  • ターミナル以外にWeb版・デスクトップ・VS Code/JetBrains・Slack等のインターフェースも提供

できないこと・前提（誤解しやすい点）

  • アカウント必須：有料サブスクリプション（Pro/Max/Team/Enterprise）または Claude Console（API・従量課金）
  • 非エンジニアの汎用PC作業向けではない（その用途はCowork）
  • npmでのインストールはNode.js 18以上が必要。ネイティブインストーラ（curl / brew / PowerShell）なら不要
  • 生成コードの正しさは保証されない。レビュー・テストは人間の責任
  • ファイルを実際に書き換えるため、Git管理下でない作業はリスクが高い

課題「空フォームでも送信できてしまう」バグ。原因箇所の特定から始めると半日かかる見込み。

シナリオB：未知のコードベースのキャッチアップとリファクタリング委任

課題引き継いだレガシーリポジトリの構造が不明。ドキュメントもなく、改修の着手点が見えない。
→
打ち手Claude Codeに構造説明と依存関係の質問を重ね、その理解の上でリファクタリングを段階委任。
→
成果物コードベースの理解（質問応答）＋async/await化などのモジュール単位の改善コミット群。

操作手順

  1. 把握フェーズ：「このプロジェクトの全体構成と主要モジュールの役割を説明して」「認証処理はどこで何に依存している？」と質問を重ねる（Claude Codeが必要なファイルを自分で読む）。
  2. 小さく委任：「認証モジュールをコールバックからasync/awaitにリファクタリングして」。1モジュールずつ依頼し、都度テストを走らせる。
  3. 記録を残す：各改善を個別コミットにさせ、最後に「直近の変更内容でリリースノートの下書きを書いて」。

注意点・つまずきポイント・コツ（Claude Code）

  • 必ずGit管理下で使う：実ファイルを書き換えるため、コミット・ブランチでいつでも巻き戻せる状態を作ってから委任する。
  • タスクは小さく具体的に：「全部直して」より「このモジュールのこの挙動を直して」。1依頼=1関心事がレビューも容易。
  • Windowsのインストールではシェルに注意：PowerShell用とCMD用でコマンドが異なる。エラー文言（irm が認識されない等）でどちらにいるか判別できる。
  • npm版が動かない場合はNode.jsのバージョン確認（18以上）。迷ったらネイティブインストーラを使う。
  • テストを「合格条件」として渡す：「テストが通るまで修正して」と依頼すると、出力品質の検証をエージェント側に組み込める。
タブ③ Claude Code — 演習 ⏱ 今日の10分

演習：自分のリポジトリを「読み取りだけ」で説明させる

ゴール：コードを1行も書き換えさせずに、Claude Codeの「コードベース理解力」だけを安全に試す。書き込みを伴うタスクは、この読み取り体験で信頼感を掴んでからでよい。

手順

  1. インストールして起動：上の「始め方」のコマンドでインストール→Git管理下の手元リポジトリで cd 対象リポジトリ && claude →初回はブラウザ認証。
  2. 読み取り系の質問を2つ投げる：「このプロジェクトの全体構成と主要モジュールの役割を説明して」→続けて「直近のコミット5件を要約して」。ファイルを手で渡す必要がないことを確認する。
  3. コマンドを2つ覚える：/help で利用可能コマンドの一覧を表示し、セッションを閉じた後に claude → /resume で会話を再開できることを確認する。

完了条件（2つチェックできたら終了）

  • 構成説明とコミット要約が、実際のリポジトリ内容に即して返ってきた
  • /help と /resume の動作を確認した

つまずいたら

  • claude コマンドが見つからない → ターミナルを開き直す。それでも不可ならPATH設定を確認、またはネイティブインストーラで入れ直す。
  • npmインストールが失敗する → node -v で18以上かを確認。満たさないならcurl/brewのネイティブ版を使う。
  • Windowsでインストールコマンドがエラー → PowerShellとCMDでコマンドが異なる。エラー文言（irm が認識されない等）で今いるシェルを判別する。

5分で終わったら追加の1手：「create a new branch called try/claude-code」→「READMEの誤字を直して」→差分確認→コミット、まで書き込み系の最小1往復を試す（ブランチ上なので安全）。

SECTION 6

まとめ：使い分けの結論（1枚）

📁 Project — 「毎回説明し直す」をやめる

同じ資料・前提で繰り返し会話するなら、最初に作るべきはこれ。ナレッジ＋指示の一度の設定が、以後すべてのチャット品質を底上げする。

🧩 Artifacts — 「会話の結果」を配れる形にする

文書・図・アプリを専用ウィンドウで反復編集し、リンク1つで公開。他人の作品もCustomizeで自分用に改変できる。

🛠 Cowork — 「相談」でなく「作業」を渡す

ローカルファイルが絡む複数ステップ作業（整理・変換・Excel/スライド作成・定期タスク）はここ。デスクトップアプリ＋有料プランが前提。

⌨ Claude Code — コードはリポジトリごと任せる

開発者向け。実装・修正・Git操作まで実行するエージェント。Coworkの土台でもある。

判断の最短経路

コードなら Claude Code → PC上の実作業なら Cowork → 繰り返す会話なら Project → 配る成果物なら Artifacts → それ以外は通常チャット。

実務では「Projectの中で会話し、成果物はArtifactsで作り、ファイル作業が発生したらCoworkに渡す」という組み合わせ運用が基本形になる。

読み終えたら：各タブ末尾の「⏱ 今日の10分」演習を1つだけ実行する。資料の理解度より、当日中に1タスク動かしたかどうかが定着を決める。
